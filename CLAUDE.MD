## RUST 编码规范
以下编码只限定本项目的编码，不针对项目生成代码的编码
1. **避免魔法值**：如果不是显而易见的平凡值，如0，1，一天有**24**小时这样的常定的数字，禁止硬编码未加说明的数字或字符串字面量，应定义具名 `const` 常量并引用。有时候，引用的库中可能已定义了相关常量，应当被优先使用。如果库中确实没有再自定义。
2. **文档化错误情况**：任何返回 `Result` 的函数都必须在文档注释中包含 `# Errors` 小节，描述可能的失败原因。
3. **优先使用原生异步模式**：优先使用 `impl Future` 或 `BoxFuture`，不要引入 `async_trait`，除非有充分且明确的理由。
4. **禁止使用 `unwrap()`**：应通过重构控制流来避免；如无法避免，使用 `expect("...原因...")` 并给出清晰的理由。
5. **保持代码可维护性**：
   - 将长函数拆分为聚焦的辅助函数
   - 为函数和变量使用有意义的名称
   - 每个函数只做一件事
   - 仅为非显而易见的逻辑添加简洁注释
   - **但不要为只被调用一次的小逻辑抽出独立的辅助方法**：抽象的成本要由复用次数或可读性收益来支付，否则内联回去比多一层间接更可读。
6. **扁平化控制流**：使用现代 Rust 风格减少嵌套。
   - 使用 `let PATTERN = expr else { ... };` 让主路径变量保持在外层作用域
   - 对于不返回值的错误分支，使用 `let Ok(value) = expr.inspect_err(|e| { ... }) else { return; };` 这类模式
   - 在 `select!` 中先返回本地 `Event` 枚举变体，再在选择之后处理
   - 善用循环返回值提升可读性
   - 始终使用 Rust 2024 的惯用写法
7. **字符串处理优先使用函数调用而非切片**：优先使用 `str::split`、`str::trim`、`str::strip_prefix` 等方法，而不是通过索引切片。例如，用 `s.strip_prefix("xxx")` 代替 `s.starts_with("xxx").then_some(&s[0..3])`，更健壮、意图更清晰。
8. **代码风格细节**（与 clippy 默认 lint 一致）：
   - **`format!` 内联变量**：能写 `format!("count={count}")` 就不要写 `format!("count={}", count)`（对应 `uninlined_format_args`）。
   - **方法引用优先于闭包**：能写 `iter.map(str::trim)` 就不要写 `iter.map(|s| s.trim())`（对应 `redundant_closure_for_method_calls`）。
9. **API 调用点的可读性**：
   - **避免 `bool` 或裸 `Option` 位置参数**：让调用方写出 `foo(false)` / `bar(None)` 这种意图不明的 callsite 是 API 设计上的失败，应优先用 enum、命名方法、newtype 等让调用点自描述。确实无法改时，在调用点用 `/*param_name*/` 注释标注实参对应的形参（字符串/字符字面量除外）。
   - **`match` 尽量穷举，避免 `_` 通配臂**：穷举写法能让新增 variant 时由编译器强制提醒所有受影响的 match 点；`_` 会让这种安全网失效。
10. 按照`<submodule>/` + `<submodule>.rs`的方式组织子mod，而**不用**`mod.rs`来组织。同时：
    - **控制单文件体积**：目标单个 `.rs` 文件 < 500 行（不含测试）；超过 ~800 行时，新功能应放进新模块，而不是继续往这个文件里加。
    - **优先私有，刻意暴露**：默认 `pub(crate)` 或私有，crate 对外的公共 API 面要明确选择、审慎暴露。
11. 不要造轮子。如果社区对于一些东西已经有了成熟的实现，那就不要自己写一个半成品。如果不知道有没有，可以想想需要类似功能的流行库，它们是依赖什么实现的。如果它们是自己实现的，考虑能否直接复用它们的代码？
12. 白箱测试：如果有需要生成白箱测试，请用submodule/tests.rs的方式，单独开一个文件，组织测试代码，不要写在主代码文件中，也不要用其它的名字（为了可维护性）。
13. 黑箱测试：如果需要生成黑箱测试，请在项目根目录下的 `tests/` 目录中创建一个新的测试文件，命名为 `<test_name>.rs`。对于需要集成测试的情况，尽量使用test-container的方式来搭建测试环境，公用的测试环境搭建方法，可以放在 `tests/common.rs` 中，供各个测试文件使用包含为子模块的方式调用。
14. **测试不要修改进程级环境变量**：测试默认并行执行，进程 env 是全局可变状态，`std::env::set_var` 会造成测试间隐式相互污染并产生偶发失败。环境派生的配置/依赖应通过参数从外部注入到被测代码中。
15. **语义正确性**：我们从可维护性与AI友好性的高度出发，高度重视代码的语义正确性。
    - 子模块的命名：要求和同级module在架构上有并列关系，并且在父module的语境下，能够简明反应该子module在其中的职责和功能。
    - 不要忽略错误：如果一个函数可能会失败，那么就应该返回错误。应避免错误被隐式转换，和其它类型的错误合并或者被吞掉。一个错误要么被明确的处理，要么就要向上传播。
    - 类型定义本身所包含信息：一个类型的定义本身就已经说明了它是怎么去看待它所组织的数据中各种关系的。也就是说，类型即文档。
16. **干净注释**：注释应该是解释代码行为，文档注释应该是说明使用方式。不要在注释和文档中引用构建计划，项目内文档，用户对话等代码使用者无法看到的内容，避免使用vN/pN/bN这样的私人约定，应该用暂时，待修复，todo等通用描述。


## 工作流程

修改代码后必须依次执行以下步骤：

1. **运行 Clippy 检查**：执行 `cargo clippy --workspace --all-targets -- -D warnings` 检查是否存在 warning，修复所有 warning 后再继续。
2. **格式化代码**：执行 `cargo fmt` 对代码进行格式化。

## 测试

- smoke 测试默认优先使用 `deepseek-v4-flash`。新增或维护真端点 smoke 时，如果场景本身不依赖更强模型，默认模型应设为 `deepseek-v4-flash`。
- 只有在场景明确依赖其它模型特性时，才切换到更高规格模型，并在对应 smoke 说明中写清楚原因。
